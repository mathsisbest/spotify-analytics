# ADR-005: Synthetic Data Fallback for Local Development

**Status:** Accepted

**Context:** The dashboard (and some ML features) depend on BigQuery queries. During local development — and for CI — BigQuery is unavailable (no GCP credentials, no project configured, or running offline). We need the dashboard to render useful content without a live database connection.

**Decision:** Use a try/except pattern with deterministic synthetic data fallback in all data-layer functions.

Every `get_*()` function in `dashboard/data.py` follows this pattern:

```python
@st.cache_data(ttl=600)
def get_top_artists(limit: int = 10) -> list[dict[str, Any]]:
    try:
        conn = st.connection("bigquery", type="bigquery")
        return conn.query(...).to_dicts()
    except Exception:
        return _synth_top_artists(limit)
```

Options considered:
- **Local BigQuery emulator** — Heavy, not officially supported by Google, and adds CI complexity.
- **SQLite fallback** — Duplicates schema in a second database. Harder to maintain.
- **Static fixture files (JSON/YAML)** — Brittle, no notion of date ranges or parameterized limits.
- **Programmatic synthetic data** — Lightweight, deterministic (seeded `np.random.default_rng(42)`), and matches the real query's return schema exactly.

**Consequences:**
- Dashboard works 100% offline — no GCP account needed for development.
- Synthetic data is deterministic — tests and screenshots are reproducible.
- Each `_synth_*` function mirrors its BQ query's column names and types — switching to real BQ is seamless.
- The try/except is intentionally broad (`except Exception`) — any failure (auth, network, missing table) falls back gracefully.
