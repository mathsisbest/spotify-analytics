# ADR-002: dbt for Data Transformation

**Status:** Accepted

**Context:** Raw data in BigQuery is ingested as-is from the Spotify API (nested JSON, inconsistent naming). Before it can power analytics and ML, it needs to be cleaned, typed, and shaped into fact + dimension tables. We need a transformation framework that is version-controlled, testable, and documents column-level lineage.

**Decision:** Use dbt (data build tool) for all data transformations.

Options considered:
- **SQL scripts in Cloud Functions** — Would require custom orchestration, no testing, no lineage, no docs.
- **Python ETL (pandas/beam on Dataflow)** — Powerful but heavyweight for this use case. Costly to run continuously.
- **dbt** — SQL-based, open-source, BigQuery-native adapter, built-in testing, documentation generation, and CI integration.

**Consequences:**
- All transformations are SQL — accessible to anyone who knows SQL.
- dbt models are organized into staging (cleaning raw data) and marts (analytical fact/dimension tables).
- Testing is declarative (not null, unique, accepted values).
- CI pipeline runs `dbt build` to validate changes before deploy.
